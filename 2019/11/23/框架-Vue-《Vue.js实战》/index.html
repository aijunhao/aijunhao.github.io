基础篇 - 第 3 章 计算属性

计算属性与 methods

计算属性对象 conputed 和方法对象 methods 主要有两个区别：

  1. 计算属性对象 conputed 中的方法不能传参，而方法对象 methods 可以。
  2. 计算属性是基于依赖缓存的，当依赖的数据发生变化的时候才会重新计算，不然就直接从缓存中拿数据返回。而方法对象每调用一次就会重新执行一次。

基础篇 - 第 7 章 组件详情（重点）

组件的功能就是将某个具体功能代码封装成组件，解决复用的问题。

组件的 data 必须是一个函数，否则会导致数据污染，在工程化中甚至会直接报错。

此外就是组件传参问题，父组件通过 props 将属性传入子组件，子组件通过 $emit(name, value) 自定义事件将数据传入到父组件并使用 @name="fun(value)" 的形式接收，或者 $on(value, fun(value))。

在子组件中接收 props 时，最好明确规定传入的类型，是否必须传入值，如果有必要可以设置默认值和验证函数。如果传入的值不能直接使用，可以用 computed 计算属性进行转换。

非父子组件通信

在非父子组件中如果需要实现通信，可以使用如下几种方法：
  • bus 中央事件总线：适合小型项目。
  • vuex 全局状态管理工具：适合大型项目。
  • this.$parent this.$children 父链和子组件索引：在子组件中使用this.$parent 可以访问该组件的父实例或组件，父组件可以通过 this.$children 访问子组件。父链和子组件有一个特点，就是可以向上或者向下无限递归访问，直到跟实例或最内层的组件。
  • localStrage 本地存储：将字段存储在本地，可以实现永久存储，任意地方访问。
  • sessionStrage ：将字段存在浏览器，在浏览器关闭前有效。

在业务中，应尽量避免父子组件数据依赖，减少父子组件耦合。

组件通信内容将在单独模块介绍，见《框架-Vue-组件通信》

slot 插槽

插槽就是一个萝卜一个坑，子组件挖坑（slot插槽），父组件填萝卜（内容）；具名插槽就是把萝卜和坑标了名称，只能对号入座，最后多的无名萝卜进无名的坑（默认插槽），没有无名的坑就丢弃萝卜，坑里有本来就有萝卜那就把萝卜拔出来填父组件的萝卜。

具名插槽

具名插槽的写法在最新的 Vue 官方文档中有些更新。

在子组件中，具名插槽在 slot 中用 name 属性指定，而在父组件中，可以用 name 属性或者 class 属性或者 v-slot 指令指定。

并且在使用上一般建议在父组件中使用 template 标签并结合 v-slot 指令使用。

基础篇 - 第 8 章 自定义指令（难点）

在一些特殊功能是，我们仍然希望对 DOM 进行底层操作，此时就要用到自定义指令。

一个指令定义对象可以提供如下几个钩子函数 (均为可选)：

  • bind：只调用一次，指令第一次绑定到元素时调用。在这里可以进行一次性的初始化设置。
  • inserted：被绑定元素插入父节点时调用 (仅保证父节点存在，但不一定已被插入文档中)。

  • update：所在组件的 VNode 更新时调用，但是可能发生在其子 VNode 更新之前。指令的值可能发生了改变，也可能没有。但是你可以通过比较更新前后的值来忽略不必要的模板更新 (详细的钩子函数参数见下)。

  • componentUpdated：指令所在组件的 VNode 及其子 VNode 全部更新后调用。

  • unbind：只调用一次，指令与元素解绑时调用。

指令钩子函数会被传入以下参数：

  • el：指令所绑定的元素，可以用来直接操作 DOM 。

  • binding: 一个对象，包含以下属性：

    • name：指令名，不包括 v- 前缀。
    • value：指令的绑定值，例如：v-my-directive="1 + 1" 中，绑定值为 2。
    • oldValue：指令绑定的前一个值，仅在 update 和 componentUpdated 钩子中可用。无论值是否改变都可用。
    • expression：字符串形式的指令表达式。例如 v-my-directive="1 + 1" 中，表达式为 "1 + 1"。
    • arg：传给指令的参数，可选。例如 v-my-directive:foo 中，参数为 "foo"。
    • modifiers：一个包含修饰符的对象。例如：v-my-directive.foo.bar 中，修饰符对象为 { foo: true, bar: true }。

  • vnode：Vue 编译生成的虚拟节点。移步 VNode API 来了解更多详情。

  • oldVnode：上一个虚拟节点，仅在 update 和 componentUpdated 钩子中可用。

除了 el 之外，其它参数都应该是只读的，切勿进行修改。如果需要在钩子之间共享数据，建议通过元素的 dataset 来进行。

拓展 - 《深入浅出 Vue.js》- 第 5 章 虚拟DOM简介

Vue.js 2.0 引入虚拟DOM ，比 Vue.js 1.0 的初始渲染速度提升 2~4倍，并大大降低内存消耗。

虚拟DOM 是 React 核心技术之一，之后被广泛运用。

是什么：虚拟 DOM 是一种将状态映射成视图的方案，与之类似的有Angular中的 脏值检测 ，Vue 1.0 中的细粒度绑定。这些方式在DOM更新时不会完全重新生成一份DOM，而是找到状态发生变化的节点，然后只更新与这个状态相关联的DOM节点。

为什么：之所以需要先使用状态生成虚拟节点，是因为如果直接用状态生成真是DOM，会有一定程度的性能浪费。

这里的状态就是指 Obejct、Number之类的数据。

怎么办：创建虚拟节点再渲染视图，就可以将虚拟节点缓存，然后使用新创建的虚拟节点和上一次渲染时缓存的虚拟节点进行对比，然后根据对比结果只更新需要更新的真是DOM节点，从而避免不必要的DOM操作，节省一定的性能开销。但是这样做是有一定代价的，当粒度太细，每一个绑定都会有一个对应的 watcher 来观察这个状态的变化，当状态被越多的节点使用时，开销就越大，对于大型项目来说，这个开销是非常大的。所以在 Vue.js 2.0 之后就使用虚拟 DOM 这种中等粒度的解决方案，组件级别是一个 watcher 实例，即便一个组件中有10个节点使用某个状态，也只有一个 watcher 在观察这个状态的变化。当状态发生变化时，通知到组件，然后组件内部通过虚拟 DOM 去进行对比与渲染。

虚拟DOM在 Vue.js 中所做的事情其实并没有想象中那么复杂，只做了两件事：

  1. 提供与真实 DOM 节点所对应的虚拟节点。
  2. 将虚拟节点 vnode 和旧虚拟节点 oldVnode 进行比较，然后更新视图。

对两个虚拟节点进行对比是虚拟DOM中最核心的算法（即 patch ），它可以判断出哪些节点发生了变化，从而只对发生了变化的节点进行更新操作。

拓展 - 《深入浅出 Vue.js》 - 第 7 章 patch 算法

源码：patch

patch 是虚拟 DOM 的核心算法，通过比较新旧两个 vnode 之间的的不同，找到需要更新的节点，然后在 DOM 上进行修改。

之所以要这么做，主要是因为 DOM 操作的执行远不如 JavaScript 的运算速度快，因此，把大量的 DOM 操作搬运到 JavaScript 中，使用 patch 算法来计算真正需要更新的节点，从而减少 DOM 操作，提高性能。

patch 对现有 DOM 进行修改需要做三件事：

  1. 创建新增的节点。
  2. 修改已经废弃的节点。
  3. 修改需要更新的节点。

patch介绍

新增节点

新增节点其实很好理解，我们有一个用最新状态创建的 vnode 节点和缓存中上一次创建的 oldVnode 节点，只要 oldVnode 中不存在，而 vnode 中存在，就需要使用 vnode 生成真实的 DOM 元素并将其插入到视图中去，然后添加到 oldVnode 作为缓存。

主要是如下两种情况：

  1. 初次渲染，DOM 中不存在任何节点，oldVnode 是不存在的，此时需要新增节点。
  2. 初次渲染后，因为状态改变而新增的节点（这里指 v-if 为 false 等一开始就不存在的节点状态变化为 true 而新增的节点），在 DOM 中并不存在，需要创建一个节点并插入到 DOM。

删除节点

如果一个节点只在 oldVnode 而在 vnode 中不存在，那个这个节点就是被废弃的节点，渲染视图时以 vnode 为准，只要 vnode 中不存在的节点就是废弃节点，需要从 DOM 中删除。

比如说：vnode 与 oldVnode 完全不同时，oldVnode 整个都需要被删除。

更新节点

更新节点指的是 vnode 和 oldVnode 是同一个节点，只是其中小部分属性发生变化，不需要完全删除旧节点新增新节点时，就需要使用更新节点。

如：新旧节点是同一个文本节点，只是节点中的文本不一样时，只需要重新设置 oldVnode 在视图中所对应的真实 DOM 节点的文本。

小结

总体来说，patch 的流程并不复杂。

创建节点

之前说 vnode 有六种节点类型，实际上真正会被创建并插入到 DOM 中的只有三种节点：元素节点、注释节点、文本节点。

三种节点比较容易区分，有 vnode 中没有 text 属性的是元素节点，isComment 属性为 true 为注释节点。

创建元素节点是一个递归的过程，通常元素节点中的 children 属性保存了当前节点的所有虚拟子节点，需要将 children 属性循环一遍，将每个虚拟子节点都创建并插入到当前节点中。



### 删除节点



从以上的源码可以看出来，其实逻辑还是比较简单的，removeVnodes() 用于删除一组节点，遍历 vnodes 节点数组，然后 isDef() 判断当前 vnode 节点是否存在不同，如果存在不同，判断 tag 标签名是否不同，如果不同，调用 removeAndInvokeRemoveHook(ch) 和 invokeDestroyHook(ch) ，否则调用 removeNode() 。

removeNode() 中的 nodeOps 是对节点操作的封装，将节点操作封装成函数放在 nodeOps 而不直接用原生的 parent.removeChild(child) 来删除节点，这涉及到平台渲染的知识，Vue.js 组件可以分别在 ios 和 Android 环境中进行原生渲染，与阿里的 Weex 类似。

所以，Vue.js 完全可以制作移动端 Web，比如开发小程序。

而跨平台渲染的本质是在设计框架的时候，要让框架的渲染机制和 DOM 解耦。只要把框架跟新 DOM 时的节点操作进行封装，就可以实现跨平台渲染，在不同平台下调节节点的操作。

removeAndInvokeRemoveHook() 移除节点并移除挂载的钩子函数。

invokeDestroyHook() 销毁钩子。

更新节点

静态节点

在更新节点前需要先判断新旧两个虚拟节点是否是静态节点，如果是，就不需要进行更新操作，因为静态节点一旦渲染到页面上之后，就不会发生状态改变。静态节点永远都不用重新渲染。

元素节点

新创建的虚拟节点没有 text 属性的就是元素节点，元素节点通常会有子节点，也就是 children 属性。

有 children 的情况

当新创建的虚拟节点 vnode 有 children 属性时，如果 oldVnode 也有 children 属性，就要进行更深入的比较并更新，在下一节详细说明。

没有 children 的情况

没有 text 属性也没有 children 属性时，表示创建的节点是一个空节点，此时会删除 oldVnode 中所有子节点，以达到空节点的目的。
