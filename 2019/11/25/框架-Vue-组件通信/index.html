框架-Vue-组件通信

前言

此篇文章主要介绍 Vue 组件通信。

组件通信主要分为父子组件通信和非父子组件通信。

父子组件通信：

  1. props : 父传子。
  2. $emit : 自定义事件，子传父。

非父子组件通信：

  1. bus 中央事件总线：适合小型项目。
  2. vuex 状态管理模式：适合大型项目。
  3. this.$parent this.$children 父链和子组件索引：适用于组件封装。
  4. webStorage 浏览器存储：localStorage 和 sessionStorage
  5. query 路由传参。

各种方式都有利有弊，下面会逐个介绍。

props

props 是父组件向子组件单向传导数据的方式。

文档参考官网：https://cn.vuejs.org/v2/guide/components-props.html

props 规范

HTML 中的特性名是大小写不敏感的，所以浏览器会把所有大写字符解释为小写字符。 这意味着当你使用 DOM 中的模板时，camelCase (驼峰命名法) 的 prop 名需要使用其等价的 kebab-case (短横线分隔命名) 命名 。

如果是字符串模板，就不存在这个限制。

props 验证

props 在接收组件中有两种写法，一种是数组，一种是对象。

通常来说，在实际项目中，props 需要明确类型，需要使用对象的写法进行验证。

自定义事件和监听

$emit 是自定义事件，在父组件中添加事件监听器，在子组件中触发自定义事件，子组件触发事件后，携带子组件参数传给监听器，即父组件。这样就实现了子组件向父组件传递数据。

$on 监听当前实例上的自定义事件，就相当于父组件中的事件监听器。

自定义事件：https://cn.vuejs.org/v2/guide/components-custom-events.html

$emit API：https://cn.vuejs.org/v2/api/#vm-emit

自定义事件名推荐使用短横线的形式。

因为 HTML 是大小写不敏感的，所以父组件中的事件监听器在 DOM 模板中会被自动转换为全小写，v-on:myEvent 会变成 v-on:myevent ，导致事件 myEvent 不会被监听到。

$on

$on API 只能监听当前实例上的自定义事件，也就是说如果是在组件化项目中，只能在子组件中监听当前组件的自定义事件。或者将子组件导入到父组件中，然后调用子组件的 $on 方法。

中央事件总线

bus 中央事件总线，是非父子组件之间的传值方式，相对于 vuex 来说较为简单，更适用于小型项目。

bus 的使用方法类似于 $emit 自定义事件和 $on 事件监听。

vue-bus 简介

vue-bus 是 npm 封装好的 bus 模块，只是对 Vue 的 $emit $on 等 API 做了简略的封装而已。当然手写原生的 Vue 实例来实现 bus 模块也非常简单。

使用

bus 只有四个 API ：

  • Vue.bus.on 或 $bus.on ：监听事件。
  • Vue.bus.once 或 $bus.once ：监听一个自定义事件，但是只触发一次。
  • Vue.bus.off 或 $bus.off ：取消监听事件。
  • Vue.bus.emit 或 $bus.emit ：自定义事件。

父链和子组件索引

在子组件中使用this.$parent 可以访问该组件的父实例或组件，父组件可以通过 this.$children 访问子组件。父链和子组件有一个特点，就是可以向上或者向下无限递归访问，直到跟实例或最内层的组件。

使用父链和子组件索引会导致组件间高度耦合，一般情况下不建议在开发中使用，当然如果是用在组件封装中，这种方式有些优势（至少不用额外导包）。

ref 属性和 $refs API

ref 属性是用来给组件指定索引名称，功能类似于 ID 或者 class 之类的选择器。

$refs 是 Vue 的 API ，用户获取指定索引名称的子组件实例或者元素，直接操作实例修改数据或者调用方法。

$ref 只在组件渲染完成后才填充，并且是非响应式的。

浏览器存储

Web Storage 浏览器存储 提供一种方式，可以让 web 页面实现在客户端浏览器中以键值对的形式在本地保存数据。html5 的 web Storage 提供两种客户端存储方式，即 sessionStorage 和 localStorage ，其目的是为了克服 Cookie 带来的一些限制。

Cookie （html4）

优点：

  1. 简单易用。
  2. 浏览器负责发送数据。
  3. 浏览器自动管理不同站点的 Cookie 。

缺点：

  1. 使用简单的文本存储数据，安全性能差，存储在本地，容易被黑客窃取。
  2. 存储大小有限，上限为4kb。
  3. 存储 Cookie 的数量有限，多数上限为30到50个，部分可以高达300个。
  4. 如果浏览器的安全配置为最高级别，使用 Cookie 会失败。
  5. Cookie 会在浏览器和服务器之间来回传递，数据量过大会导致效率低下，并且传输存在安全隐患。
  6. 生命周期为当前浏览器关闭之前所有窗口有效。

sessionStorage

优点：

  1. 存储空间大，一般是5MB，不同浏览器会有不同。
  2. 有更为丰富的接口，操作更简介。
  3. 生命周期为当前浏览器关闭之前所有窗口有效。
  4. 保存在客户端，不与服务器进行交互，节省流量，安全性相对较高。

localStorage

优点：

  1. 存储空间大，一般是5MB，不同浏览器会有不同。
  2. 有更为丰富的接口，操作更简介。
  3. 存储在本地单独文件，永久有效，除非主动删除，同源窗口有效。
  4. 保存在客户端，不与服务器进行交互，节省流量，安全性相对较高。

JSON 数据转换

在使用 sessionStorage 和 localStorage 存取数据的时候建议存取简单数据类型，此时如果是对象或者数组就需要将数据转换为 JSON 字符串，然后存储。
